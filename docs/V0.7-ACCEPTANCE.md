# LTS Agent v0.7.0 acceptance

Release acceptance completed on 2026-07-19 using a disposable clone of LBI 1.0
Build 001. The clone ran Ubuntu Server 24.04.4 LTS, kernel
`6.8.0-136-generic`, systemd 255, and x86-64.

## Release artifact

- Package: `lts-agent_0.7.0_amd64.deb`
- SHA-256: `008a579c17b8559da64dffcf7d48cc998a57d794add28ff7e1513e4e023a1f1b`
- Installed package state: `install ok installed`
- Installed account: locked `lts-agent` system user and group
- State directory: `lts-agent:lts-agent`, mode `0700`

## Acceptance results

- The authoritative privileged checks `sshd -t` and `ufw status verbose`
  succeeded; UFW was active with deny-incoming and allow-outgoing defaults.
- Corrected scripts passed all 14 validation checks and returned healthy with
  zero warnings and zero criticals as an unprivileged account.
- The packaged service reported LBI metadata, Ubuntu, kernel, `x86_64`, and
  `Africa/Lubumbashi` correctly.
- Validation was `passed`; health was `healthy`; both scripts exited zero.
- Core remained disabled and no Core state file was created.
- The absent assignment file produced the expected nonfatal warning and empty
  assignment arrays.
- The systemd timer remained enabled and active. One manual invocation and
  timer invocations at 20:28:06, 20:33:36, and 20:38:55 all completed
  successfully in under 300 ms.
- Every journal sequence showed start, completion, deactivation, and finish
  before the next activation. The service reported zero restarts and no
  remaining agent process.
- Debian install/remove/reinstall/purge tests preserved the service account and
  credential state as designed.

## Integration findings resolved

LBI 1.0 Build 001 originally used root-only SSH and UFW commands from scripts
that the agent must execute unprivileged. The compatibility patch replaces them
with observable systemd and baseline-file checks. The hardened service also
permits `AF_NETLINK`, without additional capabilities, so `ip route` can verify
the default route.

The clone's legacy `/etc/timezone` value was corrected to match its existing
`/etc/localtime` and systemd timezone configuration.

Core registration against a live LTS Core server was not part of this
acceptance; the HTTPS and synchronization paths remain covered by local TLS
tests.
