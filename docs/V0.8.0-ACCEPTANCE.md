# LTS Agent v0.8.0 acceptance

Version 0.8.0 was tested on 2026-07-20 using disposable linked-clone VM 903
from LBI 1.0 Build 002 template 901. A linked clone was used because the
Proxmox 9.1.1 `qemu-img` full-clone path reproducibly blocked while writing a
`zeroinit` LVM-thin target; template 901 remained unchanged.

- VM: `903` (`lts-agent-v080-smoke`)
- Address: `172.31.249.27`
- LBI: Ubuntu 24.04.4 LTS, version 1.0, build 002
- Package: `lts-agent_0.8.0_amd64.deb`
- Package SHA-256:
  `ab2f1e470a5bfb94d8b303072871e9c85e48f02b3a55eb8d9b78450b279dcc9e`

Cloud-init completed, hostname customization and DHCP succeeded, and the QEMU
Guest Agent was active. Privileged `sshd -t` and `ufw status verbose` checks
passed. The package installed successfully, created the locked `lts-agent`
service account, enabled the timer, and preserved stdout/stderr separation.

Direct and systemd executions ran as `lts-agent:lts-agent`, exited zero, and
reported validation `passed`, health `healthy`, Core disabled, desired state
unavailable with empty arrays, and no permission-related warning. The only
warning was the expected absence of the optional local assignment file.

Two timer activations completed at 16:47:17 and 16:49:36 CAT in 259 ms and
238 ms. Both completed before the next activation and the timer remained armed,
with no failed service result or overlap. Core-disabled execution performed no
registration, heartbeat, or desired-state request. Local TLS servers remain the
authoritative desired-state network-contract test because no LTS Core server is
implemented.

VM 100, VM 101, VM 102, template 900, and template 901 were not modified.
